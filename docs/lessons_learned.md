# GENESIS 项目经验总结

## 概述

本文档记录 GENESIS 项目开发过程中的经验教训、最佳实践和改进建议。项目从 P0 到 P8 历经约 20 周，完成了从基础设施到全链路集成的完整开发。

---

## 项目成功经验

### 1. 分层架构设计

**经验**: 三层决策架构 (L3 规划 / L2 编排 / L1 控制) 极大提升了系统的可维护性。

**具体收益**:
- LLM 规划层可独立测试和替换
- 行为树提供清晰的任务执行流程
- 底层技能可复用于不同任务

**建议**: 在类似项目中，优先考虑分层设计，每层只依赖下层接口。

### 2. 模块化测试策略

**经验**: 分级测试 (Level 1-6) 有效降低了集成风险。

**测试层级设计**:
```
Level 1: 单配方闭环 (验证基础流程)
Level 2: 双配方链式 (验证依赖处理)
Level 3: 制造 arm (验证中等复杂度)
Level 4: 完整机器人 (终极验证)
Level 5-6: 能量/故障 (边界条件)
```

**建议**: 从简单到复杂逐步验证，不要跳级测试。

### 3. 配置驱动设计

**经验**: 所有参数通过 YAML 配置文件管理，极大提升了灵活性。

**配置项示例**:
- 世界布局 (区域位置、尺寸)
- 机器人参数 (速度限制、关节限位)
- LLM 配置 (模型选择、温度参数)
- 配方定义 (输入/输出/时间/能耗)

**建议**: 尽早建立配置系统，避免硬编码。

### 4. 统一接口设计

**经验**: 每个模块提供统一接口，降低模块间耦合。

**关键接口**:
- `GenesisBot`: 机器人统一控制接口
- `StationInterface`: 工站统一交互接口
- `WorldManager.get_world_state()`: 全局状态查询

**建议**: 接口设计优先考虑上层使用便利性。

## 遇到的挑战与解决方案

### 挑战 1: IK 求解稳定性

**问题**: 初始 IK 实现在某些位姿下求解失败。

**原因分析**:
- 雅可比矩阵在奇异点附近病态
- 关节限位约束处理不当

**解决方案**:
1. 采用阻尼最小二乘法 (DLS)
2. 添加多种子求解策略
3. 实现关节限位 clamp

**代码示例**:
```python
# 阻尼最小二乘法
def solve_ik(self, target_pose, current_joints):
    for seed in self.seeds:
        result = self._solve_with_seed(target_pose, seed)
        if result.status == IKStatus.SUCCESS:
            return result
    return IKResult(status=IKStatus.FAILED)
```

**教训**: 数值算法需要考虑边界情况和数值稳定性。

### 挑战 2: 行为树状态管理

**问题**: 行为树节点间数据共享混乱，难以调试。

**原因分析**:
- 早期设计缺乏统一的数据管理机制
- 节点间直接传递数据

**解决方案**:
1. 引入 Blackboard (黑板) 模式
2. 所有共享数据通过 Blackboard 存取
3. 添加数据变更日志

**代码示例**:
```python
class Blackboard:
    def __init__(self):
        self._data = {}
        self._listeners = {}
    
    def set(self, key, value):
        old_value = self._data.get(key)
        self._data[key] = value
        self._notify_listeners(key, old_value, value)
    
    def get(self, key, default=None):
        return self._data.get(key, default)
```

**教训**: 状态管理是复杂系统的核心，需要早期设计。

### 挑战 3: LLM 响应延迟

**问题**: LLM API 调用延迟导致主循环阻塞。

**原因分析**:
- 同步调用阻塞主线程
- 网络延迟不可预测

**解决方案**:
1. 实现响应缓存机制
2. 异步调用 + 超时处理
3. 提供 Mock 后端用于测试

**代码示例**:
```python
class LLMClient:
    def __init__(self, config):
        self.cache = {} if config.enable_cache else None
    
    async def chat(self, prompt):
        cache_key = hashlib.md5(prompt.encode()).hexdigest()
        if self.cache and cache_key in self.cache:
            return self.cache[cache_key]
        
        response = await self._call_api(prompt)
        if self.cache:
            self.cache[cache_key] = response
        return response
```

**教训**: 外部服务调用需要考虑延迟和容错。

### 挑战 4: 能量平衡设计

**问题**: 初始设计中太阳能发电不足以支撑完整制造流程。

**原因分析**:
- 低估了制造能耗
- 未考虑充电效率损失

**解决方案**:
1. 重新计算能量需求
2. 增加太阳能板面积
3. 实现智能充电调度
4. 添加能量分析模块

**能量平衡公式**:
```
总发电量 = 太阳能功率 × 日照时长 × 效率
总耗电量 = Σ(任务能耗) + 充电损耗
能量比 = 总发电量 / 总耗电量 (> 1.0 表示可持续)
```

**教训**: 能量系统需要早期建模和验证。

## 技术选型反思

### 选择正确的技术

| 技术 | 选择 | 替代方案 | 评价 |
|------|------|----------|------|
| IK 算法 | 阻尼最小二乘法 | 解析 IK | ✅ 更稳定，处理奇异点 |
| 轨迹规划 | 五次多项式 | 样条插值 | ✅ 简单高效，加速度连续 |
| 任务编排 | 行为树 | FSM | ✅ 更灵活，易于扩展 |
| LLM 后端 | 多后端支持 | 单一后端 | ✅ 灵活切换，便于测试 |

### 需要改进的技术决策

| 技术 | 当前选择 | 问题 | 改进方向 |
|------|----------|------|----------|
| 感知 | 简化分割 | 精度有限 | 考虑深度学习模型 |
| 导航 | A* + Pure Pursuit | 无动态避障 | 添加 DWA/TEB |
| 装配 | 简化版 | 未真实模拟 | 添加力控装配 |

## 性能优化经验

### 1. 路径优化

**优化前**: 按任务顺序依次执行，大量空跑

**优化后**: TSP 求解器优化任务顺序

**效果**: 总移动距离减少约 30%

### 2. 并行调度

**优化前**: 等待工站加工时机器人空闲

**优化后**: 利用等待时间执行其他任务

**效果**: 总时间减少约 20%

### 3. 智能充电

**优化前**: 电量低于阈值才充电

**优化后**: 在任务间隙预充电

**效果**: 减少紧急充电中断
